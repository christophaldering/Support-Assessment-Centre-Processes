Du arbeitest an einem bestehenden Next.js 14 (App Router) SaaS-Produkt für
Executive Assessment Centers. Die Codebasis läuft, alle Features existieren.

**Deine Aufgabe ist kein Rewrite — es ist ein strukturiertes Refactoring.**

Weltklasse-Tools wie Linear, Vercel Dashboard und Raycast als Referenz hat —
nicht Applysia oder klassische HR-Software.

## 1. DESIGN-PRINZIPIEN — NICHT VERHANDELBAR

### 1.1 Command-first statt Navigation-first
- `⌘K` (Command Palette) ist der primäre Navigationsweg für Power-User
- Die Sidebar ist Orientierung, nicht Hauptweg
- Jede Aktion muss per Tastatur erreichbar sein

### 1.2 Rollen-adaptive Shell
- **NICHT:** Dieselbe App mit gesperrten Bereichen
- **JA:** Grundlegend verschiedene Shell-Konfigurationen je Rolle,
  die denselben Datenkern nutzen
- Rollen: `MASTER_ADMIN`, `WORKSPACE_ADMIN`, `MODERATOR`, `OBSERVER`,
  `PROJECT_OFFICE`, `CLIENT`, `CANDIDATE`

### 1.3 KI als Infrastruktur, nicht als Tab
- Kein Menüpunkt "Intelligence" oder "KI-Features"
- Der KI-Assistent ist ein kontextueller Panel — immer rechts, immer aktiv,
  kennt den aktuellen Assessment-Kontext
- KI-Outputs streamen (Vercel AI SDK, `ReadableStream`) — kein Spinner,
  keine Ladeseite

### 1.4 Konsistente Design-Sprache
**Farben (CSS Custom Properties — zentral definieren!):**
```css
--eds-terracotta:     #A6473B;  /* Primary Brand */
--eds-terracotta-dk:  #5F1A11;  /* Dark */
--eds-lagune:         #297587;  /* Accent */
--eds-lagune-dk:      #115560;  /* Accent Dark */
--eds-lagune-md:      #B5D6DE;
--eds-lagune-light:   #EFF4F5;  /* Light BG */
--eds-white:          #FFFFFF;
```
**Font:** Satoshi (Fontshare CDN) — bereits konfiguriert, sicherstellen dass
alle Komponenten es nutzen

**Abstände:** 4px Raster. Padding-Einheiten: 8, 12, 16, 20, 24, 32px

### 3.1 Bibliothek
Nutze `cmdk` (npm: `cmdk`). Wrap in eigene Komponente
`components/shell/CommandPalette.tsx`.

### 3.2 Trigger

- State: Zustand/Jotai-Atom `commandPaletteOpenAtom`
- Ergebnisse: API-Route `GET /api/search?q=...` mit Prisma-Fulltext

## 4. KANDIDATEN-PORTAL — BEGRIFFSKLÄRUNG & STRUKTURKORREKTUR

### 4.1 Das Problem (bisherige Benennung)
Der Begriff "Data Room" wird aktuell für **zwei völlig verschiedene Dinge**
verwendet. Das **muss** korrigiert werden:

| Bisheriger Begriff | Bedeutung | Neuer Begriff |
|-------------------|-----------|---------------|
| "Data Room" (Route /candidate/data-room) | Übungstyp: Dokumente für Fallstudie | **Fallstudie** oder **Case Study** |
| "Kandidaten-Portal" (allgemein) | Die Shell mit allen AC-Modulen | **Kandidaten-Portal** (bleibt) |

### 4.2 Was ist was — unmissverständlich

**Kandidaten-Portal** (`/portal/*` oder `/mein-ac/*`):
- Die gesamte Kandidaten-Shell
- Zeigt dem Kandidaten alle für ihn freigeschalteten Module als Kacheln
- Vergleichbar: personalisierte Startseite
- Konfiguriert vom Admin unter "Portal-Verwaltung"

**Fallstudie / Case Study** (Übungstyp `CASE_STUDY`):
- **Eine** Kachel im Portal — eine Übung unter mehreren
- Stellt dem Kandidaten ein Dokumentenpaket bereit
- Analog einem Due-Diligence-Datenraum: Unternehmensdaten, Finanzberichte,
  Strategiedokumente etc.
- Der Admin kann den Namen der Kachel frei wählen: "Fallstudie Varexia SE",
  "Unternehmensanalyse", "Strategieprojekt", etc.
- Route: `/portal/uebung/[slug]` statt `/candidate/data-room`

### 4.3 Umbenennungen im Code
```
VORHER                              NACHHER
/candidate/data-room             →  /portal/uebung/[exerciseSlug]
/candidate/data-room/[docSlug]   →  /portal/uebung/[exerciseSlug]/dokument/[docSlug]
/w/[slug]/admin/data-room        →  /w/[slug]/admin/portal/fallstudie
DataRoomCategory                 →  CaseStudyCategory (Prisma-Modell, Migration!)
PortalDocument                   →  CaseStudyDocument (Prisma-Modell, Migration!)
"Data Room" in UI-Texten         →  "Fallstudie" oder "Case Study"
```

**WICHTIG:** Datenbankmigrationen erstellen, nicht manuell umbenennen.
Prisma `renameTable` in der Migration.

### 4.4 Portal-Startseite (Kachel-Übersicht)
```
Route: /portal/home
Inhalt: Grid aller freigeschalteten Module als Apple-inspirierte Kacheln
Kachel-Typen (Enum PortalModuleType):
  - WELCOME         (Begrüßung & Ablauf)
  - PRESENTATION    (Präsentationsaufgabe)
  - ROLE_PLAY       (Rollenspiel / Mitarbeitergespräch)
  - CASE_STUDY      (Fallstudie mit Dokumentenpaket ← früher "Data Room")
  - SELF_ASSESSMENT (Selbstbeurteilung)
  - CUSTOM          (freie Übung)
```

### SCHRITT 4 — Context Panel (dynamisch) (ca. 1h)

1. Context Panel liest aktuellen Workspace-Slug aus URL
2. Lädt Assessment-Liste via `GET /api/w/[slug]/assessments` (existiert)
3. Erkennt automatisch aus der URL, welches Assessment offen ist
4. Zeigt Kontext-Unternavigation für das offene Assessment
5. Status-Dots: grün = aktiv, amber = in Vorbereitung, grau = abgeschlossen

**Abnahme:** Context Panel zeigt echte Daten, aktives Assessment
ist hervorgehoben.

### SCHRITT 5 — AI Panel (live) (ca. 2h)

1. Erstelle API-Route `GET /api/w/[slug]/ai-briefing?assessmentId=...`
   - Nutzt bestehenden LLM-Adapter (`server/llm/`)
   - Prompt: "Du bist Diagnostik-Assistent. Assessment-Daten: [JSON].
     Gib 3 konkrete Handlungsempfehlungen als JSON-Array zurück."
   - Antwortet als Stream (Vercel AI SDK `streamText`)
2. AI Panel konsumiert den Stream und rendert Insight-Karten progressiv
3. Eingabefeld sendet freie Anfragen an `POST /api/w/[slug]/ai-chat`
4. Insights sind klickbar (Action-Tags) und lösen Navigation oder
   Aktionen aus

**Abnahme:** AI Panel zeigt echte, gestreamte Insights für das
aktuelle Assessment.

### SCHRITT 6 — Begriffstrennung Data Room / Fallstudie (ca. 2h)

1. Erstelle Prisma-Migration:
   - `DataRoomCategory` → `CaseStudyCategory`
   - `DataRoomDocumentView` → `CaseStudyDocumentView`
   - Spalte `categoryId` auf `PortalDocument` bleibt, nur Relation umbenennen
2. Benenne API-Routen um:
   - `/api/candidate-portal/data-room/...` → `/api/portal/case-study/...`
   - `/api/w/[slug]/admin/data-room/...` → `/api/w/[slug]/admin/case-study/...`
3. Benenne Frontend-Routen um:
   - `/candidate/data-room` → `/portal/uebung/[slug]`
   - `/candidate/data-room/[docSlug]` → `/portal/uebung/[slug]/dokument/[docSlug]`
4. Aktualisiere alle UI-Texte: "Data Room" → "Fallstudie" wo es den
   Übungstyp meint
5. Admin-Interface: "Fallstudie-Verwaltung" statt "Data Room" in der

**Redirects einrichten** für alte URLs (Next.js `redirects` in
`next.config.js`) damit bestehende Links nicht brechen.

**Abnahme:** Alte URLs leiten um, neue URLs funktionieren,
Datenbank enthält umbenannte Tabellen.

### SCHRITT 10 — Realtime-Upgrade (Polling → SSE) (ca. 2h)

Das bestehende Polling-basierte Kollaborationssystem wird auf
Server-Sent Events umgestellt.

1. Erstelle API-Route `GET /api/w/[slug]/events` als SSE-Endpoint:
   ```typescript
   // Sendet Events für:
   // - rating_submitted (Observer hat Rating abgegeben)
   // - candidate_progress (Kandidat schließt Modul ab)
   // - observer_presence (Observer ist online/offline)
   ```
2. Client-seitiger Hook `useAssessmentEvents(assessmentId)` mit
   `EventSource`-API
3. Context Panel und Kandidaten-Liste reagieren live auf Events
4. Fallback: bei SSE-Fehler automatisch auf Polling zurückfallen

**Abnahme:** Zwei Browserfenster offen — Rating in Fenster A erscheint
ohne Refresh in Fenster B.
