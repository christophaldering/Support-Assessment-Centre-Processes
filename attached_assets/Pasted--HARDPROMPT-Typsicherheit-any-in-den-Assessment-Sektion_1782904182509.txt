DIES IST EINE VERBINDLICHE AUSFÜHRUNGSANWEISUNG, KEINE EMPFEHLUNG.

Du führst die folgenden Schritte exakt in der angegebenen Reihenfolge aus.
Bei Unklarheit: STOPP, fragen — NICHT improvisieren. Nach jedem Schritt
erstellst du exakt den angegebenen Commit. Keine Optionalität.

DIES IST EINE REINE TYPISIERUNGS-AUFGABE. ES DARF SICH KEIN LAUFZEIT-
VERHALTEN ÄNDERN — nur Typen werden präzisiert. Wenn eine Typisierung
einen bestehenden Fehler aufdeckt, der vorher durch `any` verdeckt war:
STOPP, den Fehler melden, NICHT eigenmächtig die Logik ändern.

## Ausgangslage (geprüft)

Beim Refactoring der Assessment-Detail-Seite wurden in den neuen
Sektions-Komponenten ~50 `any`-Verwendungen eingeführt, überwiegend bei
Props-Typisierung und `.map()`/`.filter()`-Callbacks. Betroffen:
- `sections/ExercisesSection.tsx` (17)
- `sections/MtmmSection.tsx` (20)
- `sections/RequirementsSection.tsx` (12)
- `sections/ObservationSheetsSection.tsx` (1)

Alle in
`app/w/[workspaceSlug]/admin/assessments/[assessmentId]/sections/`.

Die `any` zerfallen in zwei Kategorien:
- **(A) Prisma-Entitäten**: `CompetencyNode` (Felder `id`, `name`,
  `nodeType`, `sortOrder`), `ExerciseLibraryItem`, Exercise-Objekte —
  hierfür existieren generierte Prisma-Typen bzw. lassen sich lokale
  Interfaces ableiten.
- **(B) `proposal`-JSON der Anforderungsanalyse**: dynamisches JSON-Feld
  (`RequirementsAnalysis.proposal`) mit Struktur `competencies[]`,
  `assessmentModules[]`, `participants[]`, `exercises[]`, `timeline[]`,
  `adaptationNotes`, `generationPrompt`, Personen-Objekte mit
  `firstName`/`lastName`/`selected`. Hierfür werden explizite Interfaces
  definiert.

## Nicht anfassen

Laufzeit-Logik, JSX-Struktur, Handler-Implementierungen, State ·
Elternfile `page.tsx` (nur falls eine Prop-Typ-Änderung dort einen
korrespondierenden Cast erfordert — dann minimal und im Commit vermerkt) ·
andere Dateien außerhalb des `sections/`-Ordners und `sections/types.ts`.

## SCHRITT 1 — Gemeinsame Interfaces in `sections/types.ts`

In der bestehenden `sections/types.ts` folgende Interfaces ergänzen
(Feldnamen exakt aus der tatsächlichen Datennutzung im Code ableiten —
NICHT raten; wenn ein Feld im Code verwendet wird, das hier fehlt,
ergänzen; wenn unklar, ob ein Feld optional ist, als optional `?`
markieren):

1. `CompetencyNodeLike` — `{ id: string; name: string; nodeType: string; sortOrder: number }` plus alle weiteren im Code tatsächlich zugegriffenen Felder.
2. `ProposalPerson` — `{ firstName?: string; lastName?: string; selected?: boolean }` plus tatsächlich genutzte Felder.
3. `ProposalCompetency`, `ProposalModule`, `ProposalExercise`,
   `ProposalTimelineItem` — jeweils mit den im Code zugegriffenen Feldern,
   inkl. `selected?: boolean`, `adaptationNotes?: string`,
   `generationPrompt?: string` wo verwendet.
4. `AnalysisProposal` — Container-Interface mit
   `competencies?: ProposalCompetency[]`, `assessmentModules?: ProposalModule[]`,
   `participants?: ProposalPerson[]`, `exercises?: ProposalExercise[]`,
   `timeline?: ProposalTimelineItem[]`, plus weitere genutzte Arrays.
5. `LinkedAnalysis` — `{ proposal?: AnalysisProposal; ... }` mit den im
   Code zugegriffenen Feldern.
6. `ObservationSheetSection` — `{ items?: unknown[]; ... }` für die
   `sheet.content.sections`-Nutzung in `ObservationSheetsSection.tsx`.

**COMMIT (exakter Wortlaut):**
`Typen: Interfaces für Proposal-JSON und Competency-Nodes in sections/types.ts`

## SCHRITT 5 — `ExercisesSection.tsx` (17 Treffer, größte)

1. Props-`any` (`linkedAnalysis`, `libraryItems`, `specForLibrarySearch`,
   `basisExercise`, `activeModuleSpec` und deren Setter) durch konkrete
   Interfaces ersetzen — `linkedAnalysis` → `LinkedAnalysis`,
   `libraryItems` → `ExerciseLibraryItemLike[]` (Interface bei Bedarf in
   types.ts ergänzen).
2. Handler-`any` (`handleImportFromLibrary: (item: any)`,
   `handleAIVariantImport: (item: any)`) → `ExerciseLibraryItemLike`.
3. `.map`/`.filter`-Callbacks (`(m: any)`, `(ex: any)`, `(mod: any)`) →
   passende Proposal-Interfaces.
4. `(rec as any).adaptationNotes` / `.generationPrompt` → die Felder
   sind bereits in den Proposal-Interfaces (Schritt 1) definiert, daher
   kann der `as any`-Cast entfallen.
5. Für Setter, deren genaue Struktur im Elternfile liegt: falls eine
   präzise Typisierung einen korrespondierenden Typ im Elternfile
   `page.tsx` erfordert, diesen dort minimal ergänzen und im
   Commit-Text vermerken.
6. `npm run lint` für die Datei. Bei Fehler: STOPP, beheben.

**COMMIT (exakter Wortlaut):**
`Typsicherheit: ExercisesSection ohne any`
