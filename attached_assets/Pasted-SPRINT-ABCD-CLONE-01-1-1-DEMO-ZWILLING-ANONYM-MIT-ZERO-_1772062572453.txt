ZIELBILD
====================================================================

Wir benötigen einen vollständig anonymisierten, extern demo-fähigen Zwilling („ABCD“), der funktional 1:1 wie „ARAG“ ist.

- ARAG bleibt intern unverändert: /w/arag + /arag-bdp bleibt exakt wie bisher.
- ABCD ist extern demo-fähig: /w/abcd + /abcd-bdp (oder workspace-scoped Route) – ohne Kundennamen in UI/URL.
- Keine Daten-Leaks, keine Cross-Workspace Queries.
- Demo kann vollständig durchgespielt werden (Sessions, Teams, Teilnehmer, Kriterien, Bewertung, Admin, Exporte soweit vorhanden).
- „Freigabe“ erfolgt erst, wenn die QS-Checkliste zu 100% grün ist.

NICHT-VERHANDELBAR

1) „arag“ NICHT anfassen (keine Umbenennung, kein refactor, keine seed-Änderung, die arag beeinflusst).
2) Keine globale Suchen/Ersetzen-Aktion.
3) ABCD ist eigenständig:
   - eigener Workspace-Slug: abcd
   - eigene BDP Routes: /abcd-bdp (wenn BDP derzeit hardcoded auf arag-bdp)

4) Erst freigeben, wenn alle Tests bestanden sind.

SCHRITT 2 — BDP TOOL ROUTE: /abcd-bdp ALS KOPIE VON /arag-bdp
====================================================================

Da BDP aktuell „self-contained“ unter app/arag-bdp liegt, wird ein Zweitpfad benötigt.

A) Dupliziere die komplette Route:
- app/arag-bdp/**  → app/abcd-bdp/**

B) In app/abcd-bdp/* gilt:
- sichtbarer Client-Name überall: "ABCD" (nie ARAG)
- Layout-Farben bleiben gelb/schwarz
- Footer bleibt "Powered by aestimamus"

C) Auth/Session:
- Wenn BDP auth cookie bdp_session benutzt: muss identisch funktionieren.
- ABER: Bei abcd müssen DB-Lookups workspace="abcd" nutzen.
- Keine Query darf hart auf "arag" filtern.

D) Redirects:
- Login in Workspace "abcd" muss zu /abcd-bdp führen.
- /w/abcd/admin muss (wie bei arag) isoliert werden:
  Nicht-MASTER_ADMIN → redirect zu /abcd-bdp

E) Gate:
- Falls /arag-bdp/gate existiert:
  Erstelle /abcd-bdp/gate analog (redirect → /anmeldung)
  oder direkt zu /anmeldung?workspace=abcd (nur wenn existierende Parameterstruktur das zulässt, sonst nicht).

SCHRITT 7 — HARTESTE QS (MUSS 100% GRÜN SEIN)
====================================================================

Implementiere eine QS-Checkliste im Code (DEV-ONLY) UND führe manuell die Schritte aus.

A) BUILD/TYPECHECK
- npm run build (oder Replit build)
- keine TS errors
- keine lint errors (soweit im Projekt aktiv)

B) ROUTING-SMOKE
1) Öffne /anmeldung
2) Login arag (demo@demo.de/demo oder bestehende creds) → landet auf /arag-bdp
3) Login abcd (demo@demo.de/demo) → landet auf /abcd-bdp

C) ISOLATION
- Als Nicht-MASTER_ADMIN:
  /w/arag/admin → redirect /arag-bdp (bestehend)
  /w/abcd/admin → redirect /abcd-bdp (neu)
- Als MASTER_ADMIN: Zugriff bleibt möglich.

D) BDP FUNKTIONALITÄT ABCD
- /abcd-bdp lädt ohne 500er
- /abcd-bdp/sessions zeigt Sessions
- /abcd-bdp/bewertung erreichbar
- /abcd-bdp/profile erreichbar
- /abcd-bdp/admin erreichbar (als Admin)
- Admin Tabs funktionieren: Sessions/Teams/Teilnehmer/Beobachter/Kriterien/Namen/Tie-Break/Einstellungen/Export
- Keine „leer weil falscher Cookie“-Probleme: bdp_session muss für abcd funktionieren

E) DATA LEAK CHECK
- In ABCD UI darf kein Team/Teilnehmer aus ARAG auftauchen.
- In ARAG UI darf kein Team/Teilnehmer aus ABCD auftauchen.

F) STRING SCAN
- Suche zur Laufzeit (rendered UI) und im Repo:
  "ARAG" darf in app/abcd-bdp/* nicht mehr vorkommen (außer ggf. in internen Kommentaren).
- UI/Headers/Titles müssen "ABCD" anzeigen.

G) API CHECK
- Alle /api/abcd-bdp/* endpoints liefern 200 (wo relevant)
- Keine 401 bei gültiger Session
- Keine Cross-Workspace Query

H) DB CONSISTENCY
- Prüfe, dass jede ABCD Session nur ABCD Teams hat.
- Jeder Participant ist einem ABCD Team zugeordnet.
- Jede Assignment-Relation verweist nur auf ABCD.

I) PERF / UX
- Keine doppelten Nav Items
- Keine broken links
- Mobile & Desktop basic check

FREIGABE-KRITERIUM:
Erst wenn A bis I vollständig erfüllt sind, gilt Sprint als „released“.
