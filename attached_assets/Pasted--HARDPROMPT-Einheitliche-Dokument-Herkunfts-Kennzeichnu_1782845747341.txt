## Zielbild (bindend)

Jedes Dokument/Artefakt, das irgendwo in der Plattform angezeigt wird,
zeigt EINEN von drei einheitlichen Herkunfts-Zuständen an:

- **BEREITGESTELLT** — reiner menschlicher Upload, unverändert.
- **ABGELEITET** — menschlicher Upload, von der KI/System aufbereitet,
  analysiert oder transformiert.
- **GENERIERT** — vollständig KI/System-erzeugt, kein menschlicher
  Quelltext.

DIES IST AUSDRÜCKLICH KEINE SCHEMA-MIGRATION der bestehenden Datei-Felder
(`objectPath`, `sourceFilePath`, `originalFileKey`, `fileObjectPath`
bleiben UNVERÄNDERT in Namen und Bedeutung bestehen). Stattdessen wird EINE
zentrale Mapping-Schicht gebaut, die aus den bestehenden, uneinheitlichen
Feldern pro Modell den einheitlichen Status ableitet, plus EINE geteilte
UI-Komponente, die diesen Status überall gleich darstellt.

## Bindende Tatsachenbasis (geprüft, nicht zu hinterfragen)

Folgende Modelle und Felder existieren HEUTE so im Schema — du verifizierst
sie vor Beginn gegen den aktuellen Stand von `prisma/schema.prisma` und
hältst an, falls ein Feld nicht mehr existiert oder umbenannt wurde:

| Modell | relevante Felder | Pfad-Feldname |
|---|---|---|
| `Document` | (keine Origin-Felder) | `objectPath` |
| `AudioRecording` | (keine Origin-Felder) | `objectPath` |
| `Report` | (keine Origin-Felder, Modell ist konzeptionell immer Output) | `objectPath` |
| `ReportTemplate` | `isAnonymized`, `analysisStatus`, `useForStyleGuidance` (aus vorherigem Hardprompt) | `sourceFilePath` |
| `RequirementsAnalysis` | `inputType`, `transcript`, `proposal` (Json) | `objectPath` |
| `CompetencyModel` | `sourceType` | (kein Pfadfeld, Inhalt in DB) |
| `WorkspaceStyleGuide` | `sourceType` | `fileObjectPath` |
| `ExerciseLibraryItem` | `sourceContext`, `basedOnId` | `originalFileKey` |
| `ObservationSheet` | `aiGenerated` | `fileName` |
| `ObservationSheetTemplate` | `aiGenerated` | `originalFileKey` |
| `CaseStudy` | `sourceType`, `aiGenerated` | `sourceFilePath` |
| `PortalDocument` | (keine Origin-Felder) | `objectPath` |
| `Assessment`, `Exercise`, `ObserverRating`, `PredictiveProfile`, `DevelopmentBlueprint`, `DiagnosticHypothesis` | `aiGenerated` | — |

Besonderheit `RequirementsAnalysis`: EIN Datensatz enthält GLEICHZEITIG
bereitgestellten Input (`objectPath`/`transcript`) UND generierten Output
(`proposal`). Hier wird der Status NICHT auf Datensatz-Ebene, sondern PRO
ANZEIGE-KONTEXT bestimmt (siehe Schritt 2, Sonderfall).

## Nicht anfassen (absolutes Verbot, keine Ausnahmen)

Bestehende Feldnamen in `prisma/schema.prisma` (KEINE Umbenennung,
KEINE neue Migration an diesen Feldern) · bestehende Werte in
`sourceType`/`inputType`/`aiGenerated` (nur LESEN, nicht überschreiben) ·
Auth-System · RBAC · Object-Storage-Modul · `server/llm/adapter.ts` ·
sämtliche in vorherigen Hardprompts beschriebene Funktionen
(Übungsentwicklung-Umbau, Stilreferenz-Pipeline) — diese Direktive baut
NUR zusätzliche Anzeige-Logik, verändert KEINE bestehende Geschäftslogik.

Wenn eine Änderung in diesem Prompt mit einer Regel hier kollidiert: STOPP.
Konflikt melden, nicht ausführen.

## SCHRITT 1 — Zentrales Origin-Enum und Adapter

1. Neue Datei `lib/document-origin.ts`. Exportiere:
   ```
   export type DocumentOrigin = "PROVIDED" | "DERIVED" | "GENERATED";
   ```
2. In derselben Datei: EINE Adapter-Funktion PRO Modell aus der Tabelle
   oben, exakt benannt nach Muster `resolveOriginFor<Modellname>`, z. B.
   `resolveOriginForReportTemplate(template: { isAnonymized: boolean; analysisStatus: string }): DocumentOrigin`.
3. Mapping-Regeln, GENAU so umsetzen, keine eigene Interpretation:
   - `Document`, `AudioRecording`, `PortalDocument` → IMMER `"PROVIDED"`.
   - `Report` → IMMER `"GENERATED"`.
   - `ReportTemplate` → `analysisStatus === "analyzed"` UND `isAnonymized === true`
     → `"DERIVED"`. Sonst `"PROVIDED"`.
   - `CompetencyModel`, `WorkspaceStyleGuide`, `CaseStudy` (Feld `sourceType`)
     → Wert `"manual"` oder `"upload"` → `"PROVIDED"`. Wert `"ai"` →
     `"GENERATED"`. Jeder andere Wert (z. B. enthält Substring `"upload"`
     UND es existiert ein zugehöriges `aiGenerated === true`/
     `analysisStatus === "analyzed"`-Signal im selben Modell) → `"DERIVED"`.
     Trifft keine Regel zu: STOPP, exakten Wert im Code dokumentieren und
     melden statt zu raten.
   - `ExerciseLibraryItem` → `basedOnId !== null` UND `originalFileKey !== null`
     → `"DERIVED"`. Nur `originalFileKey !== null` → `"PROVIDED"`. Keins
     von beiden → `"GENERATED"`.
   - `ObservationSheet`, `ObservationSheetTemplate` →
     `originalFileKey !== null/fileName !== null` UND `aiGenerated === true`
     → `"DERIVED"`. Nur Datei vorhanden, `aiGenerated === false` →
     `"PROVIDED"`. Keine Datei, `aiGenerated === true` → `"GENERATED"`.
   - `Assessment`, `Exercise`, `ObserverRating` → `aiGenerated === true`
     → `"GENERATED"`, sonst `"PROVIDED"`.
   - `PredictiveProfile`, `DevelopmentBlueprint`, `DiagnosticHypothesis`
     → IMMER `"GENERATED"` (Default `aiGenerated: true`, kein
     Gegenbeispiel im Schema vorhanden).
4. Jede Funktion MUSS vollständig typisiert sein (keine `any`-Parameter),
   JSDoc-Kommentar mit Verweis auf die Quellfelder.

**COMMIT (exakter Wortlaut):**
`lib/document-origin.ts: zentrales Origin-Enum und Modell-Adapter`

## SCHRITT 2 — Geteilte UI-Komponente

1. Neue Datei `components/shared/DocumentOriginBadge.tsx`. Props:

2. Darstellung, GENAU diese drei Zustände, keine weiteren Varianten:
   - `"PROVIDED"` → Label „Hochgeladen", neutrale/graue Badge-Farbe.
   - `"DERIVED"` → Label „KI-aufbereitet", Bernstein/Amber-Badge-Farbe
     (konsistent mit bestehendem `ANALYSIS_BADGES`-Muster in
     `app/w/[workspaceSlug]/admin/gutachten/page.tsx` — dieselbe
     Farbpalette wiederverwenden, keine neue einführen).
   - `"GENERATED"` → Label „KI-generiert", Lila/Purple-Badge-Farbe
     (konsistent mit bestehendem `bg-purple-50 text-purple-600`-Muster in
     derselben Datei).
3. Komponente ist rein präsentational — KEINE eigene Datenabfrage, KEINE
   API-Calls. Empfängt `origin` immer von außen, berechnet via Schritt-1-
   Adapter-Funktionen am Aufrufort.
4. Sonderfall `RequirementsAnalysis`: an Anzeigeorten, die SOWOHL Input
   (Audiodatei/Transkript) ALS AUCH Output (`proposal`) gleichzeitig

   menschlichem Input basiert), eine für den Output-Teil (`"GENERATED"`,
   fest, sobald `proposal !== null`). KEIN gemeinsamer Einzel-Badge für
   diesen Datensatz-Typ.

**COMMIT (exakter Wortlaut):**

## Hinweis für künftige Hardprompts (nicht Teil dieser Ausführung)

Die tiefere Vereinheitlichung der Pfad-Feldnamen
(`objectPath`/`sourceFilePath`/`originalFileKey`/`fileObjectPath` auf
einen gemeinsamen Namen zusammenführen) ist bewusst NICHT Teil dieser
Direktive. Das ist ein separates, größeres Migrationsprojekt mit höherem
Risiko und vorwiegend internem Nutzen — wird nur auf explizite Anweisung
als eigener Hardprompt aufgesetzt.
