## Zielbild (bindend)

KEIN neues Designsystem von Grund auf. Das bestehende Token-System in
`styles/tokens.css` (Farben, Spacing, Radius, Schatten, Typografie) ist
bereits hochwertig und vollständig — es wird aber in 31 von 35 geprüften
Admin-Seiten IGNORIERT zugunsten roher Tailwind-Klassen (`slate-*`,
`gray-*`, `blue-600` statt `var(--eds-*)`). Ziel ist:

1. Konsequente Durchsetzung des bestehenden Token-Systems in ALLEN
   Admin-Seiten (keine rohen Tailwind-Farbklassen mehr außerhalb der
   semantischen Status-Farben, die bereits in `tokens.css` definiert sind).
2. EINE neue, gemeinsame Schicht ergänzen, die heute fehlt: ein
   Icon-/Illustrations-System pro Modultyp, eine `EmptyState`-Komponente,
   eine `PageHeader`-Komponente, ein Karten-Hierarchie-System
   (Primär-/Sekundär-Karte).
3. Die `ContextPanel`-Navigation (heute reine Textliste mit 20+
   gleichrangigen Einträgen) erhält Icons pro Eintrag zur visuellen
   Wiedererkennung.
4. Rollout über ALLE 31 identifizierten Admin-Dateien, in nachvollziehbaren
   Batches mit je eigenem Commit — nicht ein einzelner unüberprüfbarer
   Mega-Commit.

## PHASE 2 — Neue gemeinsame Komponenten

Alle neuen Komponenten in `components/shared/` (Ordner anlegen, falls
nicht vorhanden). Jede Komponente nutzt AUSSCHLIESSLICH `var(--eds-*)`-
Tokens, KEINE rohen Tailwind-Farbklassen.

1. **`components/shared/ModuleIcon.tsx`** — Props
   `{ type: "interview" | "presentation" | "roleplay" | "fact_finding" | "case_study" | "other"; size?: "sm" | "md" | "lg" }`.
   Rendert ein konsistentes, farbcodiertes SVG-Icon-Set (ein Icon pro Typ,
   Farbe aus `--eds-type-*`-Token aus Phase 1). Ersetzt JEDE bisherige
   Ad-hoc-Icon-/Emoji-Logik für Modultypen (insbesondere `typeIcon()` in
   `modules/page.tsx`).
2. **`components/shared/EmptyState.tsx`** — Props
   `{ icon: ReactNode; title: string; description: string; action?: { label: string; onClick: () => void } }`.
   Layout: zentriertes Icon (40px, `--eds-text-tertiary`-Farbton),
   Titel (`--eds-text-md`, `--eds-text-primary`), Beschreibung
   (`--eds-text-sm`, `--eds-text-secondary`), optionaler Button im
   bestehenden Primary-Button-Stil der Plattform.
3. **`components/shared/PageHeader.tsx`** — Props
   `{ title: string; description: string; icon?: ReactNode; actions?: ReactNode }`.
   Einheitlicher Seitenkopf: Titel in `--eds-text-3xl`, Terrakotta-Akzent
   (konsistent mit bestehendem Muster aus `gutachten/page.tsx`,
   `<h1 className="text-brand-navy">`-artige Stellen vereinheitlichen),
   Beschreibung darunter, optionale Aktions-Buttons rechtsbündig.
4. **`components/shared/Card.tsx`** — Props
   `{ variant: "primary" | "secondary"; children: ReactNode }`. `primary`
   nutzt `--eds-card-primary-shadow`/`-border`, `secondary` nutzt
   `--eds-card-secondary-shadow`/`-border`. Ersetzt verstreute,
   uneinheitliche `bg-white border border-slate-200 rounded-xl`-Stellen.

## PHASE 4 — Rollout Batch A bis E

Für JEDE Datei in JEDEM Batch (Reihenfolge A → B → C → D → E), GENAU diese
Schritte:

1. Alle rohen Tailwind-Farbklassen (`slate-*`, `gray-*`, `blue-*`,
   `red-*`, `green-*`, `amber-*` AUSSER wenn Teil eines bereits
   bestehenden, unverändert bleibenden Status-Badge-Patterns) durch
   entsprechende `var(--eds-*)`-Tokens via Inline-Style oder
   Tailwind-Arbitrary-Value (`text-[var(--eds-text-secondary)]`) ersetzen.
2. Seitenkopf (falls `<h1>`/`<h2>` + Beschreibungstext vorhanden) durch
   `<PageHeader>` aus Phase 2 ersetzen.
3. Jeden „Keine X vorhanden"/„Noch keine X erstellt"-Leerzustand durch
   `<EmptyState>` aus Phase 2 ersetzen, mit passendem Icon, Titel,
   Beschreibung UND — wo eine naheliegende primäre Aktion existiert
   (z. B. „Neu erstellen") — `action`-Prop setzen.
4. Wo Modultypen (`interview`/`presentation`/`roleplay`/`fact_finding`/
   `case_study`/`other`) dargestellt werden: `<ModuleIcon>` aus Phase 2
   statt bisheriger Ad-hoc-Lösung.
5. Karten/Listen-Container: `<Card variant="primary">` für die
   hervorgehobene Haupt-Aktion der Seite (z. B. Upload-Formular, Neu-
   Erstellen-Bereich), `<Card variant="secondary">` für nachgeordnete
   Listen/Übersichten.
6. KEINE Änderung an State, Props, API-Calls, Event-Handlern — NUR
   Markup/Klassen für Darstellung.
7. Nach JEDEM Batch (nicht nach jeder einzelnen Datei): `npm run lint`
   für die geänderten Dateien ausführen. Bei Fehlern: STOPP, beheben,
   bevor der nächste Batch beginnt.
