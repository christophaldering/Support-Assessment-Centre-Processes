# HARDPROMPT — Arbeitsbereich-Harmonisierung: Welle 0 + Welle 1 (Fundament + Pilotseiten)

DIES IST EINE VERBINDLICHE AUSFÜHRUNGSANWEISUNG, KEINE EMPFEHLUNG.

Du führst die folgenden Phasen exakt in der angegebenen Reihenfolge aus.
Bei Unklarheit: STOPP, fragen — NICHT improvisieren. Nach jeder Phase
erstellst du exakt den angegebenen Commit. Keine Optionalität.

DIES IST FUNKTIONS-ERHALTEND. Kein bestehendes Verhalten, kein API-Call,
kein State darf sich ändern — nur die visuelle Struktur wird auf ein
einheitliches System umgestellt. Wenn eine Umstellung Verhalten ändern
würde: STOPP, melden.

## Zielbild (bindend)

Aufbau eines wiederverwendbaren Layout-Systems, in das sich künftig ALLE
Content-Seiten einordnen. In dieser Welle: (0) die Fundament-Bausteine
bauen, (1) exakt VIER Pilotseiten darauf umstellen — eine pro Archetyp —
als sichtbarer Referenz-Standard. Der Vollausbau der restlichen Seiten ist
NICHT Teil dieser Welle.

## Bindende Tatsachenbasis (geprüft)

- `components/shared/` enthält BEREITS: `PageHeader.tsx`
  (`{ title, description, icon?, actions? }`), `Card.tsx`
  (`{ variant: "primary" | "secondary", children, style?, className? }`),
  `EmptyState.tsx` (`{ icon, title, description, action? }`),
  `ModuleIcon.tsx`, `DocumentOriginBadge.tsx`. Diese werden GENUTZT und
  von den neuen Bausteinen umschlossen — NICHT neu gebaut, NICHT verändert.
- `styles/tokens.css` enthält alle `--eds-*`-Tokens inkl. Spacing-Skala
  (`--eds-space-1` bis `-12`). Die neuen Bausteine nutzen ausschließlich
  diese Tokens, KEINE rohen Werte, KEINE Hex-Farben.
- Die vier Pilotseiten existieren:
  `exercise-library/page.tsx` (Archetyp 1 — Collection),
  `requirements/page.tsx` (Archetyp 2 — Master-Detail),
  `competencies/page.tsx` (Archetyp 3 — Tabs),
  `audio/page.tsx` (Archetyp 4 — Fokus-Formular).
- Es gibt AKTUELL KEINE Breadcrumb-Komponente und KEINEN einheitlichen
  Seitenrahmen. Beides wird hier neu geschaffen.

## Nicht anfassen

Geschäftslogik, State, API-Calls, Event-Handler in allen Dateien ·
bestehende `components/shared/`-Komponenten (nur nutzen) ·
`components/shell/` (linke Navigation ist fertig) · alle Seiten AUSSER den
vier genannten Pilotseiten · Kandidatenportal · `app/master/`.

Wenn eine Änderung mit einer Regel hier kollidiert: STOPP. Melden.

## PHASE 0.1 — Breadcrumb-Komponente

1. Neue Datei `components/shared/Breadcrumb.tsx`. Props:
   `{ items: { label: string; href?: string }[] }`.
2. Rendert: `Executive Diagnostics Suite › [Bereich] › [Unterseite]`.
   Letztes Item ohne Link (aktuelle Seite), vorherige als Links.
   Trennzeichen `›` in `--eds-text-tertiary`. Text `--eds-text-sm`,
   aktuelles Item `--eds-text-secondary`, Links `--eds-text-tertiary` mit
   Hover `--eds-text-brand`.
3. Rein präsentational, keine eigene Routing-Logik (Items kommen als Prop).

**COMMIT (exakter Wortlaut):**

## PHASE 0.2 — PageShell (der einheitliche Seitenrahmen)

1. Neue Datei `components/shared/PageShell.tsx`. Props:
   ```
   {
     breadcrumb: { label: string; href?: string }[];
     title: string;
     description: string;
     icon?: ReactNode;
     primaryAction?: ReactNode;   // genau EINE Primäraktion
     toolbar?: ReactNode;         // optionale Tab-/Filterleiste
     children: ReactNode;         // Inhaltszone
     maxWidth?: "standard" | "wide" | "narrow";
   }
   ```
2. Aufbau von oben nach unten, GENAU diese Reihenfolge:
   - `<Breadcrumb items={breadcrumb} />`
   - `<PageHeader title description icon actions={primaryAction} />`
     (die bestehende Komponente wiederverwenden)
   - falls `toolbar`: eine abgesetzte Leiste unter dem Kopf mit
     einheitlichem Abstand `--eds-space-6` darüber/darunter
   - `children` in einem Container, dessen Maximalbreite `maxWidth`
     bestimmt: `narrow` = `max-w-3xl`, `standard` = `max-w-5xl`,
     `wide` = volle Breite. Default `standard`.
3. Einheitliche vertikale Abstände zwischen allen Zonen über
   `--eds-space-*`. Kein seiteneigenes Spacing mehr.
4. `primaryAction` ist bewusst SINGULAR — die Ein-Aktion-Regel wird durch
   das Interface erzwungen. Weitere Aktionen gehören in ein Overflow-Menü
   (Phase 0.3), nicht als zweite Primäraktion.

**COMMIT (exakter Wortlaut):**
`Layout-System: PageShell als einheitlicher Seitenrahmen`

## PHASE 0.3 — Toolbar, ActionMenu, DataTable, CollectionGrid, MasterDetailLayout, FormSection

Alle in `components/shared/`, alle nur mit `--eds-*`-Tokens, alle rein
präsentational (Daten/Handler via Props):

1. **`Toolbar.tsx`** — `{ tabs?: {key,label}[]; activeTab?; onTabChange?; filters?: ReactNode }`.
   EIN einheitliches Design für Tab-Leiste UND Filter-Chip-Leiste. Tabs als
   Unterstrich-Tabs (aktiver Tab: `--eds-text-brand` + Unterstrich in
   `--eds-terracotta`), konsistent mit dem bereits vorhandenen Tab-Muster
   in `competencies`.
2. **`ActionMenu.tsx`** — `{ actions: {label, onClick, variant?}[] }`.
   Overflow-Menü (Drei-Punkte-Button) für sekundäre Aktionen. Erzwingt,
   dass nur EINE Primäraktion sichtbar bleibt.
3. **`DataTable.tsx`** — `{ columns, rows, rowActions? }`. Standardtabelle
   für Archetyp 1: einheitlicher Header (`--eds-bg-sunken`, `--eds-text-tertiary`,
   `--eds-text-xs` uppercase), Zeilen mit Hover, optionale Zeilen-Aktionen
   rechts.
4. **`CollectionGrid.tsx`** — `{ children }`. Responsives Karten-Raster
   (1/2/3 Spalten je Viewport) mit einheitlichem `gap` aus `--eds-space-4`.
5. **`MasterDetailLayout.tsx`** — `{ list, detail }`. Schmale Liste links
   (feste Breite, z. B. 320px), breiter Detail-/Editorbereich rechts,
   einheitlicher Trenner.
6. **`FormSection.tsx`** — `{ title?, description?, children }`. Gruppierte
   Formularsektion in einer `Card`, für Archetyp 4.

**COMMIT (exakter Wortlaut):**
`Layout-System: Toolbar, ActionMenu, DataTable, CollectionGrid, MasterDetailLayout, FormSection`
